# DSX-Connect 2 WIF / CAI Handoff - 2026-07-14

This handoff captures the current state of Google Cloud Workload Identity Federation and Cloud Asset Inventory support for DSX-Connect 2.

## Scope

The goal is to support production GCS connector deployments without long-lived Google service account JSON keys.
The target model is:

1. A Kubernetes service account identifies the connector pod.
2. GKE Workload Identity Federation maps that Kubernetes service account to a Google service account.
3. Google client libraries resolve credentials through Application Default Credentials.
4. Cloud Asset Inventory discovers buckets under a configured project, folder, or organization scope.
5. DSX-Connect 2 records which discovered buckets/scopes are protected.

## Current Implementation

The GCS connector runtime is already compatible with ADC/WIF.
If no credential path is configured, `GOOGLE_APPLICATION_CREDENTIALS` is not set and the Google SDK resolves credentials from the runtime environment.

The connector also has Cloud Asset Inventory bucket discovery:

```text
DSXCONNECTOR_GCS_ASSET_INVENTORY_SCOPE=projects/PROJECT_ID
DSXCONNECTOR_GCS_ASSET_INVENTORY_SCOPE=folders/FOLDER_ID
DSXCONNECTOR_GCS_ASSET_INVENTORY_SCOPE=organizations/ORG_ID
```

When that value is set, asset discovery can use Cloud Asset Inventory and return GCS buckets as protectable assets.

## Phase 1 Completed

The GCS connector Helm chart now supports Kubernetes service accounts for WIF:

```yaml
serviceAccount:
  create: true
  name: gcs-connector
  annotations:
    iam.gke.io/gcp-service-account: dsx-gcs-connector@PROJECT_ID.iam.gserviceaccount.com
  automountServiceAccountToken: true

gcp:
  credentialsSecretName: ""
```

Chart changes:

```text
connectors/google_cloud_storage/deploy/helm/values.yaml
connectors/google_cloud_storage/deploy/helm/templates/_helpers.tpl
connectors/google_cloud_storage/deploy/helm/templates/deployment.yaml
connectors/google_cloud_storage/deploy/helm/templates/serviceaccount.yaml
connectors/google_cloud_storage/deploy/helm/examples/values-gke-wif.example.yaml
```

The default chart behavior remains compatible with existing deployments.
By default, no service account is created and the pod uses the cluster default service account.
If `gcp.credentialsSecretName` is set, the chart still mounts the JSON key secret and sets `GOOGLE_APPLICATION_CREDENTIALS`.

## Validation Performed

Render the default chart:

```bash
helm template gcs connectors/google_cloud_storage/deploy/helm
```

Render the WIF example:

```bash
helm template gcs connectors/google_cloud_storage/deploy/helm \
  -f connectors/google_cloud_storage/deploy/helm/examples/values-gke-wif.example.yaml
```

Expected WIF rendering:

* includes a `ServiceAccount`
* includes the `iam.gke.io/gcp-service-account` annotation
* Deployment uses `serviceAccountName: gcs-connector`
* no `gcp-creds` volume is rendered
* no `GOOGLE_APPLICATION_CREDENTIALS` env var is rendered

## Important Cautions

WIF is authentication only.
It proves which workload is running, but it does not grant access by itself.
The mapped Google service account still needs IAM roles for the operations DSX-Connect should perform.

Keep JSON key support for local and lab deployments.
Use WIF as the production recommendation.

Do not collapse discovery permissions into object permissions.
A connector can discover broadly while reading or remediating only a narrower set of buckets.
